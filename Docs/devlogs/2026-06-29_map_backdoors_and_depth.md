## 1. 何を作ったか / なぜ

競合調査で確定した「メイン出入口1（Start）＋裏口1〜2／奥ほど宝・強敵」を生成器へ実装した。
- **深さメトリクス**: Start からのグラフ距離（後で宝/敵スポーンの重み付けに使う土台）。
- **裏口(ModuleRole.Exit)**: ループ生成後に残った開口のうち**最深**へ Exit モジュールを最大2個配置。

## 2. 実装の中身（原理）

- `ModuleRole.Exit` を enum **末尾に追記**（既存値の順序＝シリアライズ index を壊さない）。
- `MapPathGraph.ComputeModuleDepths(layout)`: slot 0（Start）の全ノードを起点に BFS。モジュール深さ＝
  所属ノードの最小距離。未到達は -1。決定論（列挙順非依存）。
- `SandboxCatalog`: `DeadEnd("exit", Exit)` を末尾追加（1セル・South戸口・DeadEndと同形、役割のみExit）。
- `MapGenerator`: `BackDoorCount`(既定2) を config に追加。生成順は
  **ループ閉じ → 裏口配置 → 残り開口を DeadEnd 封**。裏口は深さ降順＋座標で安定 tie-break して最深へ。

## 3. チューニングの肝（Claude が Codex 実装を調整）

裏口とループは**同じ「最深の開口」を奪い合う**（`CloseLoops` は遠いペア＝深い開口を短絡してループにする）。
ここの設計が品質を決めた:

| 方式 | 裏口0率(50seed) | ループ0率(50seed) | 判定 |
|---|---|---|---|
| 予約なし・裏口をループ前に配置 | — | **28%** | ✗ 最深の開口をループ前に潰し網目が痩せる |
| 予約なし・ループ先 | 密seedで**0個** | 12% | ✗ 高密度seedで全開口がループ化し裏口枯渇(例 conn=23) |
| 予約2・ループ先（Codex初版） | 0% | **16%** | △ 裏口必ず2だがループ閉じを早く止め網目が痩せる |
| **予約1・ループ先（採用）** | **0%** | **12%** | ✓ 裏口≥1保証・最大2、ループは導入前水準に回復 |

採用理由: ユーザー仕様は「1〜2 裏口」＝範囲（最低1）。なら**最小予約1**で「裏口≥1」を保証しつつ
最深の開口は極力ループへ回すのが仕様に最も忠実。Codex のテスト「Exit ちょうど2」は仕様より厳しすぎたため
**「1〜2」**へ是正（密seedで1個になり得る）。決定論・manifest 用途のテストの個数依存も用途本位に緩めた。

## 4. 検証（Claude 独立実施）

- `uloop compile` 0 error/warning。
- `uloop run-tests`(EditMode 全件) **66/66 緑**（深さ・裏口・決定論・manifest 往復・連結性・ループ指標）。
- 50 seed 計測: **裏口0率 0/50、ループ0率 6/50 (12%)**、裏口は1〜2（密seedで1）。seed5(conn多)も exit=2 に修復。
- 決定論・manifest checksum 不変・host/client 同一は既存テスト（ビット一致・往復）で担保。
- 制約遵守: Map 配下のみ・`[SerializeField]`/`[Header]`/`[Tooltip]` 温存・APR Player 不触・浮動小数を配置に使わない。

## 5. 残課題

- (C) 深さ→宝/敵スポーン重み付けは**ゲームプレイ層**（Treasure/敵スポーン）で後段。`ComputeModuleDepths` を消費する。
- 裏口を「最奥寄り」にもっと寄せたい場合、予約とループの配分は `reservedOpenExits`/`LoopConnections` で調整可。
- 縦・崖・起伏（モジュール焼き込み）は階層実装時。
