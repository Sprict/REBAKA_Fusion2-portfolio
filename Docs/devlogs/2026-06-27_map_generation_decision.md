# マップ生成方式の決定（手作りモジュール連結 × ホスト配布 × 自前ナビグラフ）

> **作成日:** 2026-06-27
> **公開時の状態（2026-07-22）:** この資料は実装前の設計判断です。その後、単一階層のモジュール生成、マニフェスト配布、checksum検証、自前ナビグラフを実装し、MapNetworkSandboxで2クライアントの同一再構築を確認しました。複数階層、部屋バリエーション、メインゲームへの統合、モンスターAIは未実装です。
> **決定者:** ユーザー承認済み。Claude Code（Opus 4.8）× Codex（gpt-5.5）の独立評価・反証ワークフローで導出。
> **実装後の記録:** [`2026-06-28_map_loop_quality_codex.md`](2026-06-28_map_loop_quality_codex.md)

## 0. 一行結論

REBAKA_Fusion2 のマップ生成は **「手作り部屋モジュールをシードで連結（B1/B5）＋ 内部ランダム化（D1）＋ 大構造グラフ変化」** を採用する。
同期は **ホストがレイアウトを確定し最小マニフェストを `[Networked]` 状態として配布（E2）**、地形は各クライアントが**非ネットワークでローカル Instantiate**。
モンスター徘徊は **NavMesh ではなく「モジュール埋め込みパスグラフ ＋ 局所ステアリング ＋ A*」**（ホスト権威）。
完全 procedural 生成（WFC / ノイズ / セルラー）は**不採用**。

---

## 1. 問題

リプレイ性のためマップのランダム生成が必須。だが本作は **アクティブラグドール × Photon Fusion 2 マルチ × 全部物理**という制約が重なり、「ただランダムに作る」では破綻する。決めるべきは「**どのレベルのランダム性が UX 上必要十分か**」と「**それを Fusion・物理・個人開発の制約下でどう実現するか**」。

大原則（タスク憲法）: **技術的凄さではなく UX 向上に貢献するかで選ぶ。** 「高度だから」を採用理由にしない。

## 2. 確定要件

**must:**
1. 全クライアントで完全に同一のマップ（地形・配置・経路）。手段は不問（シード決定論で各自再構築でも、ホスト生成→レイアウト配布でも可）。
   - ※当初「地形データを送らずシードのみ」と限定していたが、ユーザー確認で「同一性さえ保証されれば手段は問わない」へ緩和。これにより E2（ホスト配布）が正規候補として復活した。
2. プレイ開始時に全マップ確定（途中生成・ストリーミング不可）。
3. 個人開発で現実的に作り切れる（部屋モジュールの手作りは許容）。

**want:**
- Lethal Company レベルの「既視感が出にくい」十分なランダム性（過剰品質は不要）。
- 事前オーサリングしたナビ/ベイク資産を活かせる。
- 部屋モジュール追加だけで多様性を増やせる拡張性。
- モンスターがスタックせず徘徊できる（連結性・到達可能性の保証）。
- 範囲指定の高低差・崖を生成できる（垂直性）。
- ラグドール／グラブ純補間プロキシが生成マップ上で破綻しない。

## 4. 全候補（3次元 ＋ ナビ軸）

決定は単一選択ではなく **「生成方式 × 同期モデル × 検証保証層 × ナビ方式」** の組み合わせ。

**生成方式:** A2 複数固定マップのシード選択 / A3 固定地形＋中身ランダム / **B1 手作りモジュールのシード連結（基準線）** / B2 グリッドタイル連結 / B3 部屋＋通路グラフ(MST＋ループ) / B4 Portal制約連結 / B5 事前ベイクNavMeshモジュール連結 / C1 BSP / C2 WFC / C3 ノイズ地形 / C4 セルラーオートマトン / C5 エージェント掘削 / C6 多層3Dグリッド / C7 Heightfield / C8 ルームパッキング＋グラフ / D1 モジュール＋装飾 / D2 アルゴリズム骨格＋モジュール差し込み / D3 ミッショングラフ＋モジュール

**同期モデル（直交）:** E1 シード同期・各自決定論再構築 / E2 サーバ権威生成→配布

## 6. 決定と決定理由

### 採用: B1/B5（手作りモジュールのシード連結）＋ D1（内部ランダム化）＋ 大構造グラフ変化

- **同期 = E2 を `[Networked]` 状態として保持**（一発 RPC 不可）。payload 最小（version＋seed＋moduleList＋checksum）。**離散配置厳守**（整数グリッド座標＋90°回転＋同一prefab）で Instantiate のビット同一性を担保。
- **ナビ = N1 埋め込みパスグラフ ＋ N3 局所ステアリング ＋ A***（NavMesh 非依存）。辺に敵サイズ別の通行プロファイルを持たせ実 Collider で検証。
- **検証 = F1 連結 ＋ F5 接続口 ＋ F4 失敗時リロール（最大試行＋保証テンプレ fallback）。**
- **設計原則 = モジュールは「固定解パズル」でなく「物理サンドボックス」**（最適行動が動的文脈＝編成/運搬物/怪物位置に依存するよう設計し、単一最適解を作らない）。

**なぜこれが最良か（UXへの効き方）:**
本作の面白さの主役は **共有物理の協力ドタバタ**であり、地図は主役でなく**舞台**。B1/B5 は「導線・チョークポイント・遭遇・宝配置が毎回変わる」リプレイ性を、**連結性・公平性・物理安全な継ぎ目を常に保証しながら**提供する。`[※意見]` 「地図が地味でも物理が映える」状態こそ正解で、C系の過剰新規性は舞台を不安定にして主役（物理）を壊すので逆効果。

### 同期で E2 を E1 より優先した理由
- Codex の決定打: **「過去に StartGameArgs.Scene 未設定で RegisterSceneObjects 未発火 → 全スポーン死」「stray NetworkProjectConfig で weave 漏れ → 全スポーン死」の前例がある以上、同期の責任点（決定論再現の負荷）を増やすべきでない。** ホストが結果を配り、クライアントは「受け取った結果だけを信じる」E2 が安全。
- E1（シード決定論再構築）は離散生成なら理論上一致するが、浮動小数・PhysX・NavMeshランタイムベイク・`UnityEngine.Random`のグローバル状態・順序依存・スレッドという非決定論ハザードがあり、個人開発でクロスプラットフォーム一致を検証しきるのは脆い。
- 補足: 地形が**非ネットワーク**なので、E2 でもネットを流れるのはマニフェスト（数百バイト）だけ。地形そのものは同期しない。

### NavMesh を採らなかった理由（重要な前提反証）
- **敵AIはホスト権威**で動くため、経路探索に使うのは**ホストの構造だけ**。クライアントは敵Transformを補間描画するのみ → クライアント側のナビ構造一致は不要 → ナビ方式は決定論/同期制約から解放され、「AI品質×制作コスト」だけで選べる。
- モジュール連結生成では、ジェネレータが配置のため**モジュール隣接を既に計算している → マクロな接続グラフはタダで手に入る**。各モジュール内にノードを手置きすれば連結＝グラフ連結。決定論的・ベイク不要・設計者が通り道を制御・軽量。
- 純粋な局所ステアリング（N3単独）は凹地形/行き止まりで**局所最小に詰む** `[※理論]` ため、必ず大域構造（N1/N2）と組む。
- NavMesh（N4）は「任意点への精密ナビが本当に要る部屋」だけ後付けする**オプション**に降格。これにより F6/F7/F8（NavMeshLink・ランタイムベイク・事前ベイク連結）の難所群が消滅。

### モンスターを物理影響下に置く理由（全部物理との整合）
- テレポート/浮遊の常用は「全部物理」の信頼契約を壊すだけでなく、**プレイヤーが物理でモンスターを妨害できる**（登坂中を FF で突き落とす・積み箱を崩す）という本作 pillar のカウンタープレイを殺す。
- ただし**フル active-ragdoll AI にはしない**（不安定リスク）。「**物理相互作用する AI 制御キャラ**」= 押せる/遮れる/段差に乗る/落とせるが、意思決定・移動制御まで物理任せにしない。
- プレイヤーが動的に作る垂直ルート（積み箱・設置足場）は **スマートオブジェクト（AffordanceLink）** が設置時に動的グラフ辺を登録 → モンスターは climb/jump アクションで踏破。任意の積み箱の一般パルクール解析は研究レベルなので作らず、「箱積みは数秒の時間稼ぎ」に留める。
- 安全弁の限定ワープは「視界外・クールダウン・予告・怪物性として説明」の guardrail 付きで、長時間到達不能/技術的スタック時のみ。

## 7. 反証で見つかった破綻条件とコードベース裏取り（Phase 3）

Codex レッドチームが挙げた破綻条件を、実コードで裏取りした結果。**致命的な却下要因は出ず、すべて既知の対策で塞げる**ことを確認。最大の地雷は「E2 成功を前提に Spawn/AI/物理/ナビを一斉に始めること」。

1. **Spawn が地形生成より先に走る**（`PlayerSpawner.OnPlayerJoined` は即 Spawn・ゲート無し）→ **LayoutReady ゲート必須**（全員 Ready 後にゲームプレイ Spawn 解禁）。
2. **一発 RPC は遅参/再接続/ホストマイグレーションで取り逃す** → マニフェストは `[Networked]` 状態にし、Fusion の状態複製で自動再配信。
3. **host/client の Collider が 1cm でもズレると破綻** → 整数グリッド＋90°＋同一prefab でビット同一生成。checksum で belt-and-suspenders。client地形 = 表示＋演出専用と明文化。
4. **kinematic×kinematic グラブ破綻は既に解決済み**: `RagdollHandContact` でグラブ検出・実行は**ホスト権威**。ホスト上は手も Cube も dynamic で `OnCollisionEnter` 発火。プロキシで発火しなくても結果は `[Networked]` 同期で正しい。**地形継ぎ目とは別レイヤーで悪化しない。** 残る継ぎ目リスクは「被グラブ物/ラグドールの物理連続性（引っかかり/トンネル）」で、bridge collider＋端Collider後退＋重要物CCD＋落下リカバリで対処。
5. **静的グラフは「物理的に通れる」を保証しない**（通路幅 < 敵半径＋旋回半径 等）→ グラフ辺に敵サイズ通行プロファイル＋実Collider検証。
6. **AffordanceLink は着地点が塞がると嘘になる** → 使用直前に着地点 Overlap 検査・失敗で辺を一時無効。
7. **F4 リロール無限化**（全制約同時充足の確率が低い特定seed）→ 最大試行回数＋保証済み小テンプレへ fallback。
8. **過去事故と同種の全スポーン死リスク増**（Scene未設定/weave漏れ/manifest未受信/prefab不一致）→ **起動時検証ダッシュボード**（Scene registered / Manifest received / Prefab checksum ok / LayoutReady）。

### 7.5 「モジュール記憶 → 作業化」懸念への結論（UX反証）
プレイヤーが各モジュールの物理的最適解を覚えて作業化する懸念。結論: **完全 procedural へ寄せる理由にはならない。**
- 記憶化が痛いのは**挑戦が静的空間パズル**のとき。本作の挑戦は**動的創発**（ラグドール不安定・仲間カオス・FF・質量シェア運搬・ストーカー怪物・動的垂直）なので、既知モジュールでも「そのラン」は再生成され、**記憶化は習熟の形を変えるだけ**。
- 実証: ユーザー自身が参照点 Lethal Company（＝モジュール連結）を「既視感ほぼ無し」と評価済み。反復に感じない理由は探索順・資源・敵・音・暗さ・帰還リスク・仲間事故が毎回違うから。
- 既知モジュールは欠点であると同時に **「あの部屋だ、ここ危ない」という協力会話を生む利点**（pillar に資する）。
- 反作業化レバー（費用対効果順）: ①内部の小物/宝/罠/照明/ドア状態ランダム化（D1）②モジュールバリアント2〜4個 ③パラメトリック化 ④タイル粒度細分化 ⑤**大構造グラフ変化（Lethal型反復耐性の中心）**。①〜⑤を必要に応じて登り、⑥完全procedural へは飛ばない。

## 8. 不採用理由

- **A3 固定地形＋中身ランダム**: コア要件（リリースはランダム地形必須）を満たさない。地形固定で既視感が早い。MVP の足場としてのみ採用。
- **B3 部屋＋通路グラフ**: 真の次点。UX/同期は採用候補と同等だが、通路生成・交差・接続検証で制作コスト/実装難度が一段重い。**採用候補が動いた後の進化先**（大構造変化＝反復耐性の主軸として自然に成長）。
- **C系（BSP/WFC/ノイズ/セルラー/掘削）**: 自動生成形状は物理安定（継ぎ目でラグドール破綻）と徘徊性（連結保証）が弱く、UX目標に対し過剰品質（必要十分を超え一貫性リスクと工数を買う）。「技術的に高度だから」を採用理由にしない原則に正面から該当。完全 procedural は「将来、地形生成そのものが売りになった時の研究枠」。

## 9. 設計上の必須ガード（実装時チェックリスト）

- [ ] **LayoutReady ゲート**: 各クライアントが地形構築完了を ACK。全員 Ready 後にゲームプレイ NetworkObject を Spawn 解禁。
- [ ] **マニフェストは `[Networked]` 状態**（一発 RPC 不可）。payload は最小（version＋seed＋moduleList＋checksum）。
- [ ] **離散配置厳守**: 整数グリッド座標＋90°回転＋同一prefab。浮動小数を配置判定に使わない。専用 PRNG を生成専用に分離（`UnityEngine.Random` のグローバル状態に依存しない）。
- [ ] **非ネットワーク地形は完全静的**。状態を持つ要素（ドア/罠/壊れる床/リフト/トリガー）は NetworkObject 化 or ホスト権威コンポーネントに分離。
- [ ] **client 地形 = 表示＋演出専用**。ゲームプレイ判定はホスト結果のみ（全ラグドール物理はホスト側で走る）。
- [ ] **prefab checksum** をマニフェストに含め、不一致なら参加拒否 or 強制再ロード。
- [ ] **継ぎ目**: 接続口に共通 bridge collider、モジュール端 Collider を接続面から少し後退、重要物に CCD、落下/壁中検出でホスト側リカバリ。
- [ ] **F4 リロール**: 最大試行回数を決め、失敗時は保証済み小テンプレへ fallback（無限ループ/開始遅延の防止）。
- [ ] **ナビ**: グラフ辺に敵サイズ通行プロファイル。AffordanceLink 使用直前に着地点 Overlap 検査。
- [ ] **ワープ guardrail**: 視界外・非戦闘・安全地点・ログ出力・クールダウン。
- [ ] **起動時検証ダッシュボード**: Scene registered / Manifest received / Prefab checksum ok / LayoutReady を一画面で確認。

## 10. 自力再実装チェックリスト（原理理解の確認）

1. なぜ地形を NetworkObject にせず非ネットワークのローカル Instantiate にするのか説明できる（壁を全部 NetworkObject 化する無駄を避ける／流れるのはマニフェストだけ）。
2. E1 と E2 の違いと、本作で E2 を選んだ運用上の理由（同期の責任点を増やさない・過去の全スポーン死）を説明できる。
3. E1 の非決定論ハザード（浮動小数/PhysX/NavMeshベイク/グローバルRandom/順序/スレッド）を列挙し、離散配置で回避する原理を説明できる。
4. 「全ラグドール物理はホスト側で走る（Gang Beasts方式 `if(!HasStateAuthority)return`）」ことが、client 地形ズレが物理破綻を起こさない根拠であると説明できる。
5. NavMesh を使わずモジュール埋め込みパスグラフ＋局所ステアリング＋A* で徘徊を成立させる構成と、ホスト権威がクライアント側ナビ一致を不要にする理由を説明できる。
6. 局所ステアリング単独が局所最小で詰む理由と、大域グラフと組む必要性を説明できる。
7. プレイヤーの動的垂直（積み箱/設置足場）に AffordanceLink（スマートオブジェクト）で追従する仕組みと、一般パルクール解析を作らない判断を説明できる。
8. モンスターを「物理相互作用する AI 制御キャラ」にする理由（公平性＋カウンタープレイ＝pillar）と、フル ragdoll AI にしない理由（不安定）を説明できる。
9. 「モジュール記憶→作業化」懸念に対し、本作の挑戦が動的創発であること・反作業化レバーの順序を説明できる。
10. LayoutReady ゲートが無いと何が壊れるか（地形未生成の上に Spawn）を説明できる。

## 11. MVP 順序

1. **A3 相当**（単一/少数固定マップ）で抽出ループ（運搬・救出・脱出）を検証。
2. **B1**（5〜8モジュール: Start/Goal/Straight/Corner/T/LargeRoom/Stairs/DeadEnd ＋ ソケットスナップ ＋ F1連結 ＋ 埋め込みグラフ ＋ LayoutReady ゲート ＋ マニフェスト `[Networked]` 配布）。一本道＋短い分岐から。
3. **D1 装飾＋モジュール追加**で多様性。モジュール内部ランダム化（小物/宝/罠/照明/ドア）。
4. （任意）**B3 グラフ・AffordanceLink・縦モジュール**拡張。モンスター AI（パスグラフ＋ステアリング）導入。

## 12. 残課題・将来拡張

- **5b（残課題）**: NavMesh 非依存はポートフォリオで「なぜ標準 NavMesh を使わない?」と問われうる。採用理由（垂直・生成・軽量グラフ・ホスト権威）の説明準備＋失敗ログ可視化を用意する。平面モジュールだけ NavMesh 併用のハイブリッドも選択肢。
- モンスターのワープ「物理純度」の線引き（純度最優先 / guardrail付き怪物性ワープを設計組込み / 中間）は、敵設計フェーズで確定する（マップ生成決定には非依存）。
- 大構造グラフ生成（B3）への進化、鍵ゲート/ミッショングラフ、ループ通路、縦穴/梯子/エレベーター、モンスター巡回 Waypoint レイヤー。

## 付記: 評価手法について（ポートフォリオ観点）

本決定は「Claude（コードベース密着）×Codex（ゼロベース独立＋レッドチーム反証）」を**互いの結論を隠して**走らせ、5回の独立評価がすべて同方向へ収束したことで信頼度を担保した。確証バイアスを避けるため、推し案を**能動的に否定**させ、出た反論を**実コードで裏取り**した。この「独立二者の突き合わせ＋反証駆動」の意思決定プロセス自体が、技術選定の説明力を示す実例になる。
