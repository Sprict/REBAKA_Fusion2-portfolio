# ダメージシステム設計決定（2026-07-06）

> 種別: 設計判断（実装前）。コード変更なし。
> 経緯: ユーザー原案 → Claude レビュー → 対話で論点を1つずつ確定。

## 1. 問題 / 目的

「ダメージも全部物理で決める」というゲームコンセプトに沿ったダメージシステムを設計する。
要件:

- プレイヤー含む全オブジェクトが物理衝突でダメージを受け、耐久値0で破壊/死亡する
- 立つ・運ぶ・引きずるは無傷。転倒で頭を打つ・高所落下・罠・落下物ではダメージ
- 剣は切るたびに消耗する（武器＝消耗資源）
- エネミー剣戟は主役にしない（コア＝アクティブラグドールと物理で罠/エネミーを回避しながら宝物運搬。武器無双でホラー感が消えるのを避ける）
- 理不尽な死は嫌（ただし対策手段は必ず残す）

## 2. 決定事項

### 2.1 ダメージ式（確定）

```
接触ペアの鋭さ S = max(自分の鋭さ, 相手の鋭さ)
各オブジェクトが受けるダメージ = J × S ÷ 自分の硬度
（J = 衝突力積。Unity の Collision.impulse を使用）
ダメージ < 1.0 は無傷（閾値）
```

- **力積ベース**: 原案の「質量×速度」は「どちらの質量/速度か」が物理的に曖昧で、絶対速度だと並走接触で誤ダメージが出る。[※理論] 衝突の実効的な強さは相対速度と換算質量で決まり、Unity はそれを計算済みの `Collision.impulse` として提供する。自前計算より正確で実装も楽。
- **鋭さは接触ペアで共有（max）**: 刃(5)×体(1)なら双方に S=5 を適用。剣も切るたびにダメージを受ける（＝消耗する）が、剣は**硬度が高い**ので受けるダメージは小さい。「鋭いものは双方向に危険、硬いものは長持ち」が1行の式で成立。
- **閾値はダメージ計算後に適用**: 副作用として「鋭い物は軽く触れただけで閾値を超える」＝**置いてある剣の刃に寄りかかると怪我をする世界**になる。レビューで顕在化した副作用だが、コンセプトに合うので意図的に採用。
- **同一接触ペアのクールダウン必須**（目安 0.2〜0.5 秒）: OnCollisionStay は接触継続中毎物理ステップ発火するため、閾値ギリギリの押し付け合いで毎ステップ削れるのを防ぐ。

### 2.2 max vs sum の比較（sum 不採用の理由）

検討した対立案: `S = 自分の鋭さ + 相手の鋭さ`。

| 接触ペア | max | sum |
|---|---|---|
| 体 × 地面 | 1 | 2 |
| 刃 × 体 | 5 | 6 |
| 刃 × 刃 | **5** | **10** |

実質差は「刃×刃」のみ。sum は武器同士の激突を最大の破壊イベントにし、「峰で受けろ」等の防御部位選択の駆け引きを生むが、**このゲームは剣戟を意図的に外す**ためその場面がほぼ存在しない。max は「この接触の危険度＝関わる中で一番鋭いもの」と常に一言で説明でき、**ルールの可読性＝死の納得感**（理不尽な死の回避）に直結する。将来迷ったら `S = max + k×min`（k=0でmax、k=1でsum）で連続調整可能だが、最初から一般形は実装しない。

### 2.3 表面プロパティと耐久値の分離

- **表面プロパティ（鋭さ・硬度）**: 全コライダーが持つ。コライダー単位で異なってよい（剣＝刃5/柄1、地形＝岩場1.5/滑らかな床1.0 のように地形の危険度も表現）。
- **耐久値**: 壊れうるものだけが持つ。地面・壁には**付けない**（「ほぼ無限の耐久値」ではなくコンポーネント自体を省く）。地形との接触はダメージ計算を半分スキップでき、接触イベント大量発生するこのゲームでは性能面でも有利。

### 2.4 プレイヤー

- 耐久値は**プレイヤー全体で1つ**。部位ごとに持つのは不採用（「腕だけ壊れたら？」等の定義コストが高い）。
- 部位ごとに**硬度**が異なる（頭＝硬度低＝同じ衝突で大ダメージ）。「頭を打つと痛い」はこれで十分表現できる。
- 将来の防具システム＝部位の硬度を上げる、として自然に拡張できる。

### 2.5 閾値の較正方法

体感基準から逆算する。[※理論] 高さ h からの落下速度は v = √(2gh)。「頭（鋭さ1同士）は 2m 落下からダメージが入り始める」のような基準を先に決め、そこからダメージ 1.0 の物理スケールを逆算 → 他オブジェクトの数値も芋づる式に決まる。

### 2.6 エネミー

- **倒せる存在にする**。ただし「武器で簡単に倒せる」は式ではなく**ステータスで防ぐ**: 硬度・耐久値を十分高く設定 →「剣で斬っても手応えがないが、巨大な岩の下敷きにすれば倒せなくはない」。物理の創意工夫だけが報われる＝「対策はできるが理不尽ではない」。倒す難度はパラメータ2つで連続調整可能。
- **リスポーン: コスト容量式（Lethal Company 参考）**。マップごとにコスト容量があり、時間経過で容量が増え、容量を埋めるようにスポーン。キルが買うのは「恒久的安全」ではなく「一定時間の安全地帯」。容量の時間増加は人工的な制限時間なしで「早く帰れ」圧力を生み、宝物運搬ループと相性が良い。コスト上限＝エネミー同時数上限＝**NetworkObject 数の上限をデザイン側で保証**（同期コスト管理の実利）。
- **スポーン地点は物理的な実体**（穴・扉・裂け目）として配置。リスポーンが「あそこからまた来た」という説明可能な現象になる。プレイヤーが物でスポーン地点を塞げる → バリケードは**本ダメージシステムでエネミーが物理破壊してくる**（仕組みの掛け算、追加実装ほぼゼロ）。
- **予告（テレグラフ）**: [※理論] 脅威の事前知覚可能化は「理不尽な死」を「自分のミス」に変換する汎用原則（LC のダクト足音、L4D の音楽、DRG のアナウンス）。本作では物理言語に翻訳する: スポーン予定の裂け目から小石が物理的に転がり落ちる等、**世界が物理的に予兆を見せる**。
- **背後湧き禁止ルールを明示すること**（プレイヤー視界内・一定距離内にはスポーンしない）。コスト式の代表的な理不尽ポイントへの対策。
- メカニクスの参考は法的にも文化的にも問題なし（[※理論] ゲームメカニクスは著作権保護対象外。保護は表現）。R.E.P.O 自体が「物理強化 LC ライク」で高評価。差別化点は「全部物理」。

### 2.7 ネットワーク方針

- **ホストが衝突からダメージを計算**し、結果の耐久値（float 1個）だけを `[Networked]` で同期。衝突イベント自体は一切レプリケーションしない。
- [※理論] `[Networked]` は変化時のみ差分送信されるため、通信コストはダメージ発生瞬間に数バイト/オブジェクト。既存の transform 同期（毎tick）に比べ誤差レベルで、**4人プレイ・オブジェクト数の制約要因にならない**。オブジェクト数上限は従来通り物理 transform 同期側が決める。
- クライアントプロキシは kinematic 純補間（既存方針）なのでクライアント側では衝撃が取れない＝ホスト権威が事実上の一択、という前提とも整合。

## 3. 不採用案まとめ

| 案 | 不採用理由 |
|---|---|
| ダメージ＝質量×速度（原案） | どちらの質量/速度か曖昧。絶対速度だと並走接触で誤ダメージ。エンジンが正確な力積をくれる |
| S = sum（鋭さの和） | 差が出るのは刃×刃のみで、剣戟を外す本作では発揮場面がない。可読性で max が勝る |
| 部位ごとの耐久値 | 部位破壊の定義コストが高い。硬度差で「頭は痛い」は表現できる |
| 地形に無限耐久値 | 耐久値コンポーネント自体を省く方がシンプルかつ高速 |
| エネミー無敵（地形扱い） | 「原理上は倒せるが実用的でない」の方が創意工夫の余地を残す。倒しても補充されるので無菌化しない |
| 式でエネミー無双を防ぐ | 対称な式のままステータス（硬度・耐久）で調整する方が単純で連続調整可能 |
