## 1. プロジェクト概要

### 1.1 基本情報

- **タイトル**: REBAKA_Fusion2（仮）
- **ジャンル**: アクティブラグドール協力抽出ローグライト（Lethal Company に近い構造、操作系はラグドール）
- **技術スタック**: Unity 6000.3.7f1, Photon Fusion 2.1.0
- **プレイヤー数**: 2〜4 人スケーラブル（First Playable / MVP Core は 2 人、MVP Complete 以降で 2〜4 人）
- **フレンドリーファイア**: あり（pillar の中核を構成）
- **想定 1 ラン時間**: First Playable 1〜3 分 / MVP 5〜8 分 / 本番 15 分前後

### 1.2 ポートフォリオとしての位置づけ

このプロジェクトは**ポートフォリオ作品**でありつつ、**作者が本当に作りたい題材**であることを両立する。各設計判断に対して「なぜそうしたか」を一貫して語れることを優先する。

技術的な焦点:
1. **ネットワーク物理同期の技術理解**: 多関節ラグドールを 4 人で同期する Tick rate / Interest Management の実装と挑戦
2. **ゲームデザインの一貫性**: 単一の design pillar から派生原則を導き、すべての要素を pillar に貢献させる思考
3. **問題解決能力**: バグや課題を発見し、根本原因を理解して解決するプロセス
4. **AI との協働力**: AI を道具として活用し、自分で理解を深める姿勢

## 2. Design Pillar（最重要・確定）

### 2.1 Pillar

**「ラグドールとしての協力の物理」**

この pillar に貢献しない要素は容赦なく削る。pillar からの派生原則:

- 宝物は**物理オブジェクト**（重さ・サイズ・両手持ち・落下時の挙動）
- トラップは**物理連携で解く**もの（一人がスイッチ、二人で運搬など）
- モンスターは**物理的に絡む**存在（掴む・引きずる・投げる）
- 死亡しても**ラグドール身体は物理として残り続ける**（仲間が引きずって脱出 → 救出成功）
- 4 人協力は「役割分担の必然性」が物理から生まれる

### 2.2 Pillar の見せ場（4 大シーン）

1. **物理運搬** — 重い宝を 2〜3 人で抱えて運ぶ。一人が転ぶと巻き込まれる
2. **ダウン救出**（死体運搬）— 罠で倒れた仲間（HP 0、ラグドール脱力）を引きずって出口まで運ぶ＝救出成功
3. **拘束救出**（救出スクラム）— Bracken に掴まれた仲間（意識あり、操作制限）を、複数人で押す/引く物理力で剥がす
4. **物理パス** — 断絶を越えるために宝を仲間に投げ渡す

これら 4 シーンは最終的な MVP の見せ場だが、最初から同時に作らない。

実装順:
1. **First Playable** — 物理運搬だけを 2 人で成立させる
2. **MVP Core** — 固定 2 部屋の 1 ランに、ダウン救出を追加する
3. **MVP Complete** — 拘束救出と物理パスを最低実装で追加し、4 大シーンが 1 ラン中に発火し得る状態にする
4. **Portfolio Target** — 4 人プレイ、Room Kit の部屋数、演出、計測、説明資料を磨く

> **用語整理**: 本作には 2 種の「救出体験」がある。**ダウン救出**は §3.3、**拘束救出**は §3.4 で詳述する。

## 3. ゲーム設計

### 3.1 1 ランの流れ（MVP）

First Playable:
```
入口 / テスト廊下
   ↓
重い宝を 2 人で運ぶ
   ↓
抽出ゾーン
```

MVP Core / MVP Complete:
```
入口（ロビー部屋）
   ↓
固定 2 部屋 → Room Kit 抽選で 3〜5 部屋（罠 / 宝室 / 通路 / モンスター部屋）
   ↓
出口（抽出ゾーン）
```

- First Playable はランダム抽選なし。`Test_Playground` 相当の固定構成で、物理運搬の手触りだけを見る
- MVP Core は固定 2 部屋で 1 ランを成立させる
- MVP Complete で Room Kit 抽選を導入する
- 1 ラン 5〜8 分（MVP Complete）を目標

### 3.2 マップ生成（簡易 PCG: Room Kit 抽選方式）

完全 PCG は採用しない（理由: ネットワーク同期という技術的挑戦の軸と分離した別問題になり、scope が爆発する）。

採用方式:
- 手作り部屋テンプレ（MVP Complete は 3〜5 個、Portfolio Target は 5〜10 個程度）+ ランダム抽選 + 接続トポロジ
- 部屋接続は事前定義のドア/通路ピボットで整合
- **インクリメンタル方針（必須）**: 1 テスト廊下 → 2 部屋固定 → 3〜5 部屋抽選 → 5〜10 部屋へ拡張
- 抽選導入は MVP Complete 以降。First Playable と MVP Core では固定構成を優先する

部屋テンプレの設計指針（pillar 起点）:
- 重い宝を 2 人で運ぶ廊下
- 一人がスイッチを踏み続ける間、他が宝を回収する罠部屋
- 倒れた仲間を引きずって脱出する必要のあるダウン罠
- 宝を仲間に投げ渡さないと越えられない断絶

### 3.3 デス処理（救出モデル）

- HP 0 でラグドール完全脱力（操作不能）
- 仲間が足/手を持ち上げて引きずれる（既存の物理コンタクト系を流用）
- 出口まで運べば救出成功 = 救助賞与スコア
- ラン終了時に救出されていなければ「行方不明」扱い

→ 「死体運搬」自体が pillar を体現する**最大の見せ場**になる。死亡が単なるペナルティでなく、co-op 体験の中心に組み込まれる。

### 3.4 モンスター設計（Bracken ハイブリッド型）

Lethal Company の Bracken をベースに「掴み + 引きずり + 救出可能」を統合した 1 種を MVP Complete で投入する。

#### 最終形のコアループ
1. プレイヤーが視界に入ると停止、外れると加速して背後接近
2. 接触すると IK で腕を絡め、プレイヤーを「拘束状態」に
3. 最寄りの罠（または部屋ごと固定の引きずり地点）に向かってドラッグ
4. 仲間が押す/引く物理力で掴みを引き剥がせる（= 救出機会）

#### Pillar 貢献
- 「背後を仲間が見張る」役割分担が**自然に発生**
- 「掴まれた仲間を物理で剥がす」体験が**死体救出と双璧の見せ場**
- 1 体の AI 設計に投資する見返りが大きい（緊張感 + 救出機会の二刀流）

#### 実装スコープ（段階化）

Bracken は MVP の中でも重い要素なので、段階的に実装する。

| 段階 | 目的 | 実装内容 | 明確にやらないこと |
|------|------|----------|--------------------|
| B0: 拘束プロトタイプ | 救出スクラムの成立確認 | 接触したら拘束状態にする / 部屋内の固定引きずり地点へ力をかける / 仲間の接触または一定入力で解除 | IK、NavMesh、視線 AI、複雑な経路探索 |
| B1: 視線・距離ループ | Bracken らしい緊張の追加 | 視界に入ると停止、外れると接近 / 部屋ごとの固定引きずり地点を使う | 本格 Behavior Tree、複数敵 |
| B2: 物理救出 | pillar の見せ場化 | 仲間からの押す/引く力または接触人数で解除しきい値を満たす | 細かいアニメーション品質 |
| B3: 演出 polish | 動画や公開時に見せやすくする | 腕 IK、SE/VFX、カメラ演出、チューニング | 新しい敵種 |

MVP Complete の合格ラインは B1〜B2。B3 は Portfolio Target。

### 3.5 メタ進行（MVP は 1 ラン完結）

- 永続アンロック / ショップ / スキルツリーは **MVP 対象外**
- 将来的にはクォータ制（LC 型）導入も視野（現時点では設計のみ）
- 優先順: まず「1 ランが楽しく回る手触り」を確立してから拡張判断

→ MVP がブレないし、「**意図的に scope を絞った理由**」を一貫して語れる。

### 3.6 人数設計（2〜4 人スケーラブル）

- First Playable / MVP Core は 2 人で最低限動かす（テスト容易）
- MVP Complete では 2〜4 人に対応し、4 人での同期負荷と役割分担を検証する
- Portfolio Target では 4 人時に「役割分担の必然性」が最も光る状態を目指す
- 人数が少ないほど難しい（重い宝、複数スイッチが負担）= 自然な難易度カーブ
- デモでは「2 人でも遊べる最低体験」と「4 人で技術的に光る完成形」を分けて説明する

## 4. LC（Lethal Company）との明確な差別化

| 軸 | LC | 本作 |
|----|----|------|
| 操作キャラ | 標準的なヒト型 | **アクティブラグドール（ConfigurableJoint）** |
| 死亡時 | 死体は動かない / 持ち帰り可 | **物理ボディとして引きずれる、救出可** |
| 怪物の脅威 | 攻撃で即死（Bracken は首をへし折る） | **掴まれて引きずられる、仲間が物理で剥がせる** |
| 重い物の運搬 | スタミナで遅くなる | **複数人で支え合う物理運搬** |
| 宝物 | アイテム扱い | **物理オブジェクト（重心・両手持ち）** |
| 中核体験 | 恐怖と発見 | **協力の物理コメディと緊張** |

差別化軸の核は「**物理が co-op の文法そのものになっている**」点。LC が「移動 + 探索 + アイテム回収」だとすれば、本作は「物理の押し合い圧し合いで全部表現する」。

## 5. MVP 範囲

### 5.1 マイルストーン定義

以後、単に「MVP」と書く場合は **MVP Complete** を指す。ただし実装は必ず First Playable から進める。

| 段階 | 目的 | 含めるもの | 含めないもの | 合格ライン |
|------|------|------------|--------------|------------|
| First Playable | 物理運搬の核を最速で検証する | 2 人 / 固定テスト廊下 / 宝 1 種 / 抽出ゾーン / デバッグ HUD | Room Kit、罠、死亡、Bracken、物理パス | 2 人で重い宝を運び、抽出ゾーンまで持ち帰れる |
| MVP Core | 1 ランの最小ゲームループを作る | 2 人 / 固定 2 部屋 / 宝 1〜2 種 / 罠 1 種 / ダウン救出 / 簡易スコア | Room Kit 抽選、Bracken、複数罠、メタ進行 | 物理運搬とダウン救出が 1 ランで成立する |
| MVP Complete | 本仕様の pillar 見せ場を一通り発火可能にする | 2〜4 人 / Room Kit 3〜5 部屋 / 罠 1〜2 種 / Bracken B1〜B2 / 物理パス部屋 / 結果画面 | ショップ、永続アンロック、複数敵、長時間ラン | 4 大シーンが最低実装で 1 ラン中に発火し得る |
| Portfolio Target | 動画や公開時に強く見せる | 4 人 / Room Kit 5〜10 部屋 / 罠 2〜3 種 / Bracken B3 / 同期計測 / polish | スコープ外のメタ進行拡張 | 「なぜこの設計か」を実演付きで説明できる |

### 5.2 MVP から外すもの

- 永続アンロック / ショップ / スキルツリー
- 複数モンスター種
- 完全 PCG
- 本格的な Behavior Tree
- 大量の部屋テンプレ
- ゲームパッド最適化

これらは pillar の検証後に判断する。First Playable と MVP Core では、まず「物理運搬が面白いか」「救出が co-op の中心になるか」を見る。

## 6. 既存資産との親和性

### 6.1 活かせる資産（Synergy）

- **Photon Fusion 2.1.0**: 4 人協力に直結
- **APR ベースのアクティブラグドール**（20+ ファイルに責務分離済み）: pillar の物理基盤そのもの
- **物理トライアド + ネットワークペアの安定化**: 救出モデルと運搬の前提
- **既存の `RagdollImpactContact.cs` / `RagdollFootContact.cs`**: 掴み/接触判定の拡張ポイント
- **`SyncMetricsRecorder`** ほか診断系: 4 人同期負荷の計測に直接使える

### 6.2 新規開発が必要な領域

- Room Kit 抽選 PCG（ジェネレータ + 接続トポロジ）
- 宝物の物理運搬（両手持ち / 投げ渡し / 重さ表現）
- 救出モデル（ダウン状態 + 仲間による引きずり）
- Bracken ハイブリッド AI + 掴み + 救出スクラム
- 罠 2〜3 種

### 6.3 ネットワーク権限モデル（source of truth）

基本方針:
- **Input Authority** はプレイヤー入力だけを送る
- **State Authority** がゲーム状態、物理的な成否、スコアを決める
- クライアント側の UI / ラベル / IK / VFX は presentation として扱い、勝敗や拘束の真実源にしない
- 既存の player authority 実装と衝突する場合は、既存実装を確認してからこの表を更新する

| 対象 | State Authority が決めること | Networked にするもの | presentation 扱いでよいもの |
|------|------------------------------|----------------------|------------------------------|
| 宝物 Treasure | 掴み受理 / 掴み解除 / 現在質量 / 抽出成功 | `GrabberCount`, `CurrentMass`, 抽出済み状態 | デバッグラベル、掴みエフェクト |
| プレイヤー / ダウン身体 | HP 0 判定 / ダウン状態 / 救出成功 / 復帰可否 | 生死状態、ダウン状態、救出済みフラグ | 脱力の見た目補間、救出 UI |
| Bracken | AI 状態 / target / 拘束開始 / 拘束解除 / 引きずり地点 | AI state、target id、拘束状態、drag point id | 腕 IK、演出用の揺れ、SE/VFX |
| 罠 | 発火判定 / ダメージ / down 付与 / reset | trap state、cooldown、hit result | ローカル警告表示、演出 |
| Room Kit | seed / 採用部屋 / 接続順 / 出口位置 | run seed、room template ids、connection ids | マップ表示、部屋名ラベル |
| スコア / 結果 | 持ち帰り宝 / 救出賞与 / 行方不明判定 | score、rescued players、missing players | 結果画面アニメーション |

「入力は各 player、成否は State Authority、見た目は各 client」という分離で説明できる。

## 9. 設計判断の一貫性（why の整理）

### 9.1 デザインの why

- **なぜ Photon Fusion?** → 多関節ラグドール（〜10 ConfigurableJoint）を 4 人で同期する Tick rate / Interest Management の挑戦
- **なぜラグドール?** → pillar「協力の物理」の物理的根拠。HFF や LC では実現しない「死体運搬」「救出スクラム」が出せる
- **なぜ Room Kit 抽選?** → 完全 PCG はネットワーク同期と分離して語れない別問題。リプレイ性は確保しつつ scope を抑えた判断
- **なぜ救出モデル?** → 「死」をペナルティでなく pillar 体験に変換する設計。死体が物理として残る → ラグドール基盤の必然性が出る
- **なぜ Bracken ハイブリッド?** → 1 体の AI に 2 つの pillar 見せ場（緊張感 + 救出スクラム）を凝縮、コスパが高い
- **なぜ MVP でメタ進行を切ったか?** → 「一周遊べる手触り」を最初に確立するのが co-op 体験の質を見るうえで必須。クォータ等は後で追加可能
- **なぜ MVP を段階化したか?** → 4 大シーンを同時に作ると scope が爆発するため、まず物理運搬、次に救出、最後に Bracken / 物理パスを足す
- **なぜ 2 人から始めるのか?** → テスト容易性を優先し、最低体験を 2 人で成立させる。4 人は同期負荷と役割分担を見せる Portfolio Target として磨く

### 9.2 技術理解（v2.0 から継続）

1. NetworkRigidbody はどういう仕組みで動いていますか？
2. State Authority とは何ですか？
3. 物理演算の同期で一番苦労した点は？
4. ネットワークラグの影響をどう対処しましたか？
5. Interest Management が必要になるのはどんな時か？

## 10. 旧 v2.0 仕様の取り扱い

v2.0 (2026-01-30) は「Human Fall Flat 系の物理挙動デモ + 2 人 + ダンジョン無し」だった。これは短期間で技術的な完成度を示せる状態を作るための極限までスコープを切った構成だった。

v3.0 への移行で v2.0 のうち **継続するもの**:
- アクティブラグドール基盤
- Photon Fusion 採用
- フレンドリーファイア
- 「実装の完成度よりも技術理解の深さ」という基本姿勢

v3.0 で **拡張・変更するもの**:
- プレイヤー数 2 → 2〜4
- ダンジョン無し → Room Kit 抽選 PCG
- 物理デモのみ → 抽出ゲームループ + モンスター + 救出モデル
- 「物理挙動を見せる」 → 「物理を協力の文法にする」

v2.0 の正本テキストは Git 履歴 (commit `2550c5f` 以前) に残る。本ファイルが以後の単一の正本（source of truth）。
