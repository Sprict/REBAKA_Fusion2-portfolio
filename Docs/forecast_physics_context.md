# REBAKA ネットワーク物理設計 — Forecast Physics評価まとめ

## プロジェクト概要

**REBAKA** は最大4人協力のPvEアクティブラグドールゲーム。洞窟ダンジョンでパッシブラグドールの敵と戦いながら、Lethal Companyのようにアイテムを持ち帰る。フレンドリーファイアあり（GangBeasts的なカオス）。

- **ネットワーク**: Photon Fusion 2.1プレビュー（Forecast Physics使用）
- **対戦形式**: 最大4人PvE協力（PvPではない）
- **ターゲットping**: ~100ms（国内）
- **1キャラクターあたりのRigidbody数**: 10〜20個
- **敵**: パッシブラグドール（モーター制御なし、ホスト管理）

---

## 最優先の設計条件

1. **テレポートしない・振動しない**（最優先）
2. 入力遅延は許容
3. クライアント間の多少の位置ズレは許容

---

## Forecast Physicsの評価結果

### ✅ 適合する点

- **ロールバックなし** → 原理的に振動しない
- **外挿+補間** → スナップしない
- **入力遅延は設計上存在する** → ゲームデザインで吸収可能
- **全クライアントで同一入力から同一モーター計算を実行する設計** → アクティブラグドールと相性が良い
- **FixedUpdate分裂は「問題」ではなく「Forecast Physicsの意図した設計」**

### ⚠️ 残るリスク

- **Fusion 2.1プレビューAPIの変更リスク**（プロトタイプなら許容範囲）
- **殴った/殴られたタイミングでのAuthority設計**（後述の通り、PvE設計で大幅に軽減）

## セカンドオピニオンで判明した重要事項

### Authority問題は当初の評価より大幅に軽い

ドキュメントは「プレイヤー同士が直接衝突するバトルゲーム」を前提にAuthority問題を最大リスクとしていたが、REBAKAの実態はPvE協力ゲーム。

- **主要ゲームループ**: 殴る側＝プレイヤー（アクティブラグドール）、殴られる側＝敵（パッシブラグドール、ホスト管理）→ Authority切り替え不要
- **Authority切り替えが必要なケース**: フレンドリーファイア時、アイテム受け渡し時のみ → 頻度が低く、多少雑な実装でもOK
- **アイテム運搬**: Forecast Physicsサンプルの「距離ベースAuthority切り替え」がほぼそのまま使える

### 新たに浮上した本当のリスク

1. **パッシブラグドール敵の同期帯域がボトルネックになりうる**
   - 各敵も10-20個のRigidbodyを持つ場合、4プレイヤー＋複数敵で同期帯域がかなり厳しい
   - Forecast Physicsの外挿は帯域削減に寄与するが、同時存在ラグドール数に上限を意識する必要あり

2. **PhysXの非決定論性によるドリフト**
   - 同じ入力・同じモーター計算でも、クライアント間で浮動小数点の評価順序やソルバー差でフレーム毎にドリフトが蓄積する可能性
   - 「姿勢が自然に揃う」は楽観的すぎる可能性があり、補正の頻度・強度次第
   - ただし100ms/PvE/GangBeasts的カオスなら許容度は広い

3. **PvEでの外挿誤差許容度は広い**
   - 競技PvPではないので「仲間のラグドールが50msぶんズレている」は体験を壊しにくい
   - GangBeasts的カオス感がむしろプラスに働く

## Forecast Physicsの正しいアーキテクチャ（REBAKAの場合）

```
【プレイヤーキャラクター（アクティブラグドール）】

全クライアントで実行:
  FixedUpdate() {
    // モーター制御（全クライアントで同一計算）
    joint.targetRotation = ComputeTargetRotation(networkInput);
  }

StateAuthority（ホスト）のみ:
  FixedUpdateNetwork() {
    // 状態キャプチャ（NetworkedプロパティはFusionが管理）
  }

【敵キャラクター（パッシブラグドール）】

ホストのみ:
  FixedUpdate() + FixedUpdateNetwork() {
    // 物理シミュレーション + NetworkTransformで同期
  }

他クライアント:
  // Forecast Physicsの外挿+補間で表示
```

## 結論と推奨される次のステップ

**Forecast PhysicsはREBAKAに最も整合性が高い選択肢。** PvE設計のおかげでAuthority問題は大幅に軽く、最優先条件（テレポートしない・振動しない）を原理的に満たす。

**理論整理はこれ以上深掘りしても収穫が薄いフェーズ。** 次は最小構成プロトタイプで実機検証すべき。

### プロトタイプで検証すべき項目

1. **振動しないか** — アクティブラグドールが外挿+補間で安定して表示されるか
2. **帯域は現実的か** — プレイヤー2人＋パッシブラグドール敵2体で帯域を計測
3. **PhysXドリフト** — クライアント間で姿勢がどの程度ズレるか、補正の挙動
4. **アイテムピックアップ** — Authority切り替え時の見た目の自然さ
5. **フレンドリーファイア** — プレイヤー同士の衝突時にワープ/スナップが発生しないか
