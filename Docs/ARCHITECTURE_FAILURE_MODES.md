# Bungling Delvers — 現在の制約と未検証事項

> **最終確認:** 2026-07-22
>
> 過去に発生した不具合、現在も残る制約、まだ検証していないリスクを区別して整理します。

## 読み方

- **解決済み**: 原因を切り分け、現行方式へ反映済み
- **現在の制約**: 現行実装で意図的に受け入れている、または未解決の範囲
- **未検証**: 起こる可能性はあるが、実機結果がなく断定できない項目

## 解決済み

### クライアント側ラグドールの振動・発散

旧方式では、クライアントが一部の受信ポーズを追いながら残りの部位をローカル物理で動かしていました。描画フレーム間の補間もなく、ホストとの小さな差が四肢のずれやハードスナップへ発展していました。

現在は次の方式です。

1. State Authorityだけがプレイヤー本体の物理結果を確定する
2. Rootと、Root相対の14部位を毎tickの確定ポーズとして複製する
3. クライアントは本体15パーツをkinematicにし、受信済みの2つのポーズ間を`Render()`で補間する

判断と実装の記録は[`2026-03-27_syncmetrics_baseline_measurement.md`](devlogs/2026-03-27_syncmetrics_baseline_measurement.md)と[`2026-06-10_client_snapshot_interpolation.md`](devlogs/2026-06-10_client_snapshot_interpolation.md)にあります。

### 掴み処理の権限不一致

以前はクライアント側の接触検出からRPCでState Authorityへ掴みを要求していました。現在は、入力の取得、接触判定、ジョイント生成と破棄をState Authority上にそろえています。現行の`RagdollHandContact`は掴み用RPCを使用しません。

詳細は[`2026-06-18_peer_sync_pure_interpolation.md`](devlogs/2026-06-18_peer_sync_pure_interpolation.md)を参照してください。

### 切断後にプレイヤーやRunnerが残留する問題

現在は、切断時にロビーへ安全に戻す処理、退出プレイヤーのDespawn、ActivePlayersと一致しないオブジェクトのcleanup、親階層から外れたラグドールRootの破棄を実装しています。

これは「同じ試合を継続できる」という意味ではありません。ホスト終了後の扱いは次節の制約に残ります。

## 現在の制約

### ホスト終了後は同じセッションを継続できない

Host Migrationと試合状態の復元は未実装です。ホストが終了した場合、クライアントはロビーへ戻れますが、進行中のセッションを別ホストへ引き継ぐことはできません。

このプロジェクトは現在プロトタイプであり、専用サーバー、永続バックエンド、公開サービス向けの認証基盤は対象外です。

### プレイヤーと一部の物理オブジェクトに表示上の隙間が残る

プレイヤーとCubeは、どちらもホスト権威ですが異なる補間パイプラインで描画されます。その実効遅延差により、クライアント画面では押している手とCubeの間に隙間が見える場合があります。

掴み・運搬・解放と衝突結果はホスト側で成立していますが、この相対表示のずれは未解決です。調査結果は[`2026-06-18_peer_sync_pure_interpolation.md`](devlogs/2026-06-18_peer_sync_pure_interpolation.md#7-その他の残課題フォロー)に記録しています。

### 実行時設定の一部は公開抜粋に含めていない

公開リポジトリは応募向けのコード抜粋です。シーン、Prefab、Photonのアプリ設定、実行時に使用するProfileアセットの一部は含めていません。そのため、公開コードだけでビルドを再現することは目的としていません。

## 未検証

### 3〜4人同時接続時の負荷

全身ポーズ同期のデータ量は概算していますが、3〜4人が同時に衝突・掴み・移動する条件でのCPU負荷、帯域、missing input、長時間安定性は未計測です。「4人で飽和する」とは現時点では断定できません。

### 長時間・不安定回線での耐性

基本的な2クライアント接続、切断、再参加は確認しています。一方で、長時間セッション、継続的なpacket loss、大きなjitterを組み合わせた試験は未実施です。

### 公開サービスとしてのセキュリティ

現在の検証は開発用セッションを前提としています。第三者が参加する公開サービスとしての認証、入室制御、運用監視、侵入試験は実施していません。したがって、この資料では盗聴やgriefingを確認済みの障害として扱いません。

## 更新方針

この資料では、観測済みの不具合と将来のリスクを混同しません。検証結果が増えた場合は、条件、結果、証拠となるdevlogを追加して項目を更新します。
