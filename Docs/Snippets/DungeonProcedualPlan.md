カテゴリA: グリッド/タイルベース アルゴリズム

A-1. BSP（Binary Space Partitioning）木

- 原理: 2D空間を再帰的に二分割→末端パーティションに部屋配置→隣接部屋を廊下で接続
- 長所: 部屋の重なりなし、構造的、高速、決定的
- 短所: 矩形部屋ばかりで人工的になりがち
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: 50〜150行（20〜30時間）
- 採用ゲーム: Nethack、多数のローグライク
- 参考: https://eskerda.com/bsp-dungeon-generation/

A-2. セルラーオートマトン（CA）

- 原理: ランダムグリッドに隣接セルルールを反復適用→有機的な洞窟形状が自然発生
- 長所: 自然な洞窟風、チューニング可能
- 短所: 「部屋＋廊下」構造になりにくい、後処理必要
- 適合: マルチプレイ★5 / モジュラー3D★3（洞窟向き）
- 実装量: 60〜200行（15〜25時間）
- 採用ゲーム: Brogue
- 参考: https://blog.jrheard.com/procedural-dungeon-generation-cellular-automata

A-3. ランダムウォーク（酔歩）

- 原理: ランダム方向に進みながら床を敷く、複数ウォーカーで分岐
- 長所: 極めてシンプル（20行）、蛇行する通路が自然
- 短所: 偏りやすい、明確な「部屋」は生成しにくい
- 適合: マルチプレイ★5 / モジュラー3D★3
- 実装量: 20〜50行（5〜10時間）

A-4. Wave Function Collapse（WFC）

- 原理: タイル間の隣接制約を定義→エントロピー最小セルから確定→制約伝播→全セル確定
- 長所: 制約に従った一貫性のあるレイアウト、非常に柔軟、サンプル入力から学習可能
- 短所: 実装が複雑、矛盾で失敗する可能性
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: 200〜400行（30〜50時間）
- 参考: https://github.com/mxgmn/WaveFunctionCollapse, https://github.com/oddmax/unity-wave-function-collapse-3d

A-5. Perlin Noise + 閾値

- 原理: 2D Perlin Noise生成→閾値以上＝壁、以下＝床→後処理で部屋・廊下抽出
- 長所: 非常に高速、自然な形状
- 短所: 細かい構造制御が困難、洞窟向き
- 適合: マルチプレイ★5 / モジュラー3D★2
- 実装量: 30〜100行（5〜15時間）

カテゴリB: 部屋＋廊下 アプローチ

B-1. ランダム部屋配置＋廊下接続

- 原理: ランダム位置に矩形部屋配置（重なり拒否）→部屋中心を廊下で接続→全到達可能を保証
- 長所: 明確な部屋＋廊下構造、部屋タイプ設定可能
- 短所: チューニング不足だと不自然
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: 100〜200行（20〜35時間）

B-2. BSP + 部屋 ハイブリッド

- 原理: BSPで空間分割→各リーフに1部屋→パーティション境界で隣接部屋接続
- 長所: BSPの構造性＋部屋の明確さの両立
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: 150〜250行（25〜40時間）

B-3. Delaunay三角分割 + 最小全域木（MST）

- 原理: 部屋中心でDelaunay三角分割→MSTで最小接続→一部非MST辺をループ用に追加
- 長所: 有機的な部屋分布、数学的に健全、廊下生成がエレガント
- 短所: 2D平面的、3D拡張は非自明
- 適合: マルチプレイ★5 / モジュラー3D★4
- 実装量: 幾何アルゴリズム含め（25〜40時間）
- 採用ゲーム: TinyKeep
- 参考: https://www.gamedeveloper.com/programming/procedural-dungeon-generation-algorithm

B-4. Voronoi図による部屋形状生成

- 原理: ランダムシード点でVoronoi分割→各セルが有機的な部屋形状→Delaunayの双対で接続
- 長所: 非矩形の有機的な部屋形状、自然な空間分布
- 短所: モジュラープレハブとの相性が限定的
- 適合: マルチプレイ★5 / モジュラー3D★2（不定形のため）
- 実装量: （25〜40時間）

カテゴリC: グラフベース アプローチ

C-1. ミッショングラフ文法

- 原理: ダンジョンを有向グラフ表現→グラフ書き換えルールで展開→空間レイアウトに変換
- 長所: 複雑なミッション設計可能、ロジックとジオメトリの分離
- 短所: 学習コスト非常に高い、デバッグ困難
- 適合: マルチプレイ★5 / モジュラー3D★4
- 実装量: 500〜1000行以上（60〜100時間）
- 採用ゲーム: Unexplored
- 参考: https://www.boristhebrave.com/2021/04/02/graph-rewriting/

C-2. Lock-and-Key パターン

- 原理: 進行グラフ生成→鍵配置で探索順序を強制→他アルゴリズムと組み合わせ
- 長所: 自然なプログレッション、パズル要素
- 短所: バランス調整が必要
- 適合: マルチプレイ★4 / モジュラー3D★3
- 実装量: 150〜300行（20〜35時間）
- 参考: https://www.boristhebrave.com/2021/02/27/lock-and-key-dungeons/

カテゴリD: プレハブ/モジュールベース

D-1. ソケット/スナップ システム

- 原理: 部屋プレハブに「ソケット」（接続ポイント＋メタデータ）→互換ソケットマッチングでモジュール配置
- 長所: 最大限のアート品質保持、高速、モジュラー
- 短所: 部屋バリエーション事前制作が必要
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: 配置ロジック100〜250行（25〜40時間）

D-2. Spelunky式テンプレートグリッド

- 原理: NxNグリッド→各セルに事前定義した部屋テンプレートをランダム選択→接続口を合わせる
- 長所: 最もシンプル、高速、品質保証
- 短所: グリッド感が強い、テンプレート量が必要
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: 100〜200行（10〜15時間）
- 採用ゲーム: Spelunky

D-3. Binding of Isaac式（BFS拡張 + 特殊部屋配置）★注目

- 原理: 開始部屋からBFS拡張（隣接2未満制約）→目標部屋数まで→ボス＝最遠、宝物＝末端、秘密＝交差点付近→部屋内装はテンプレートプールから選択
- 長所: シンプルかつ実績あり、部屋タイプの配置ルールが明確、ローグライクに最適
- 短所: グリッド整列、限定的なバリエーション
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: （25〜40時間）
- 採用ゲーム: The Binding of Isaac
- 参考: https://www.boristhebrave.com/2020/09/12/dungeon-generation-in-binding-of-isaac/

D-4. Dead Cells式（手作り部屋 + グラフレイアウト）★注目

- 原理: デザイナーが大量の部屋テンプレート制作（各入口/出口メタデータ付き）→アルゴリズムが抽象グラフ生成→各ノードに互換テンプレート選択
- 長所: 最高の視覚品質（手作り）、プレイヤーに「デザインされた」感覚
- 短所: テンプレート制作の工数が大きい
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: コアアルゴリズム簡単、テンプレート制作が主工数（30〜50時間）
- 採用ゲーム: Dead Cells, Hades（ハイブリッド）
- 参考: https://www.gamedeveloper.com/design/building-the-level-design-of-a-procedurally-generated-metroidvania-a-hybrid-approach-

D-5. Enter the Gungeon式（品質重視ループ接続）

- 原理: 全部屋を個別にプレイテスト→接続グラフで中央に密なループ→部屋タイプ分散（戦闘、廊下、コネクタ）
- 長所: 最高品質のペーシング、探索の満足感
- 短所: 非常に高い制作工数
- 適合: マルチプレイ★5 / モジュラー3D★5
- 実装量: （40〜60時間 + コンテンツ制作）
- 採用ゲーム: Enter the Gungeon
- 参考: https://www.boristhebrave.com/2019/07/28/dungeon-generation-in-enter-the-gungeon/

カテゴリG: 迷路ベース

G-1. DFS再帰バックトラック迷路

- 原理: グリッド上でDFS→壁を掘りながら進む→完全迷路生成→一部壁を除去してループ追加
- 長所: 数学的にエレガント、長い探索パス、シンプル

- 長所: 均等な分岐、バランスの良い探索
- 短所: 同上（部屋なし）
- 適合: マルチプレイ★5 / モジュラー3D★3
- 実装量: 50〜100行（10〜20時間）

カテゴリI: 形式言語/制約ベース

I-1. L-System（リンデンマイヤーシステム）

- 原理: 再帰的文法ルールで文字列変換→空間構造にマッピング（例: Room→Room+Corridor+Room）
- 長所: 制御可能、階層的構造の表現に強い
- 短所: 文法パーサー+空間ソルバーが必要
- 適合: マルチプレイ★5 / モジュラー3D★4
- 実装量: （40〜60時間）

I-2. Answer Set Programming（ASP）/ 制約充足問題

- 原理: ダンジョン生成を制約充足問題として宣言的に定義→ソルバーが有効な構成を探索
- 長所: 複雑な設計ルールを表現可能（ボス前に宝物必須等）
- 短所: 外部ソルバー必要、学習コスト高
- 適合: マルチプレイ★5 / モジュラー3D★4
- 実装量: 制約設計が大変（60〜100時間）

I-3. マルコフ連鎖（部屋シーケンス生成）

- 原理: 部屋タイプの遷移確率行列を定義→チェーンを歩いて部屋シーケンス生成→空間配置は別アルゴリズム
- 長所: センスのある部屋順序（連続ボス部屋回避等）、軽量
- 短所: シーケンスのみで空間レイアウトは別途必要
- 適合: マルチプレイ★5 / モジュラー3D★4（補助的に使用）
- 実装量: （15〜25時間）
- 参考: Dead Cellsが部屋タイプ選択に使用

補助システム（メイン手法と組み合わせて使用）

S-1. Poisson Disk Sampling（小道具配置）

- 用途: 部屋内のTorch, Barrel, Pot等の自然な配置
- 原理: 最小距離制約付きランダム点生成→重なりのない自然な分布
- 実装量: 2〜5時間
- 採用ゲーム: Dead Cells, Hades（装飾配置に使用）

S-2. 難易度グラデーション

- 用途: スタートからの距離に応じて敵・罠の強さを段階的に上昇
- 原理: Dijkstraでスタートからの距離計算→距離ベースで難易度スコア割当
- 実装量: 20〜35時間

S-3. バイオーム/テーマゾーン

- 用途: ダンジョン内の視覚・難易度バリエーション
- 原理: 部屋をゾーンに割当→ゾーン毎に異なるテンプレートプール・テクスチャ・敵を使用
- 実装量: 25〜40時間
- 31プレハブでの分類例: 石造り系(10) / 下水道系(5) / 装飾・特殊系(16)

S-4. 多階層（マルチフロア）

- 用途: 階段で接続された複数フロアのダンジョン
- 原理: 各階を独立生成→特定部屋間に階段配置→上階の天井確保
- 実装量: 35〜50時間
- Stairプレハブ: 利用可能（Gridnessパックに含む）

S-5. マルチプレイ決定的生成（必須）

- 用途: 全クライアントが同一ダンジョンを生成
- 原理: ホストがシード値を送信→全クライアントが同一RNGで生成
- 注意: C#のRandom.Rangeはプラットフォーム間で非決定的→MT19937等の自前RNG必須
- 実装量: 10〜20時間
- 参考: [※理論] Photon FusionのNetworkRNGを使えばFusionが決定性を保証する可能性あり（要確認）

推奨実装案（4案、詳細版）

案1: BSP + モジュラー自動配置（バランス型・推奨）

- レイアウト: BSP木で部屋配置決定
- 壁/床配置: 部屋形状からRule Tileロジックで自動プレハブ選択
- 小道具: Poisson Disk Samplingで部屋タイプに応じた装飾配置
- GUI: EditorWindow（サイズ、シード、部屋タイプ比率、生成ボタン）
- 工数: 約30〜45時間
- 技術的挑戦度: ★5（アルゴリズム＋ツール＋マルチプレイ）

案2: Binding of Isaac式 + テンプレートプール（ローグライク最適）

- レイアウト: BFS拡張でグリッド上に部屋配置、特殊部屋ルール適用
- 部屋内装: テンプレートプール（部屋タイプ×入口構成で選択）
- 小道具: テンプレート内に事前配置 + ランダム追加
- GUI: EditorWindow（フロアサイズ、部屋数、シード、テンプレート管理）
- 工数: 約25〜40時間
- 技術的挑戦度: ★4（実績あるゲームデザインパターンの理解）

案3: WFC制約ベース（先端技術型）

- レイアウト: 各プレハブに隣接ルール定義→WFCが制約を満たすレイアウト生成
- 壁/床/柱: 全てWFCの制約で統一的に処理
- GUI: EditorWindow + 制約ルール可視化エディタ
- 工数: 約40〜60時間
- 技術的挑戦度: ★5（最先端アルゴリズム知識）

案4: Dead Cells式ハイブリッド（品質最優先型）

- レイアウト: 抽象グラフ生成→手作りテンプレートマッチング
- 部屋: 手作りプレハブテンプレート（入口/出口メタデータ付き）を多数制作
- 小道具: テンプレート内に手動配置で最高品質保証
- GUI: EditorWindow（テンプレート管理+グラフ生成パラメータ）
- 工数: 約40〜60時間（テンプレート制作が主）
- 技術的挑戦度: ★5（プロダクション品質のゲームデザイン理解）

検証方法

1.  EditorWindow から「生成」→ダンジョンが正しく生成されること
2.  同じシード→同じレイアウト再現（決定性検証）
3.  全部屋が到達可能（接続性検証 - BFS/DFS）
4.  壁・床・ドアの隙間やめり込みなし（視覚検証）
5.  マルチプレイ: ホスト/クライアントで同シード→同ダンジョン
6.  30秒以上の物理安定性確認（ラグドール物理との干渉チェック）
